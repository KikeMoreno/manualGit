Flujos de GitHub

Un flujo de trabajo o Workflow es una forma de organizar las tareas de forma que se conozca de antemano,
que tareas prosiguen y cual es la meta o destino. Con ello, evitamos conflictos y que finalicemos con un trabajo más predecible.

1. Centralized Workflow

Centralized Workflow o Flujo de Trabajo Centralizado, consiste en un único repositorio como punto central para todos los cambios del proyecto.

Primeramente, clonamos el proyecto del repositorio de GitHub en nuestro repositorio local. Modificamos el proyecto y una vez finalizado, realizamos un commit. Finalmente, todos los cambios que queramos subir a GitHub, los desarrolladores deberán cambiar a la rama master y realizar un push.

a. Posibles errores
Imaginemos que dos desarrolladores, Kevin y John se han descargado la última actualización del proyecto, y que Kevin empieza modificar los archivos y realiza un push en origin/master.
Por otro lado, John también modifica el proyecto e intenta hacer otro push en GitHub, sin embargo, no puede hacer el commit ya que no tiene el proyecto actualizado. De este modo, si John quiere subir sus cambios, primeramente deberá actualizar su repositorio y seguidamente, introducir sus cambios. También puede realizar git pull –rebase origin master.

2. Feature Branch Workflow

La idea principal es que en este flujo de trabajo, cada apartado se desarrolle en una rama diferente a la master. Una característica de este modelo son los pull request. Éstos servirán especialmente para iniciar discusiones sobre el proyecto en cuestión.

Feature Branch Workflow consiste en un repositorio central con una rama master que continua siendo la rama oficial del proyecto. Pero, a la hora de hacer un commit directamente en la rama local master, los desarrolladores crean nuevas ramas para tratar cada apartado.

a. Pull request
Es una petición de un desarrollador o usuario hacia otro, para que este último incorpore los commits. Cuando finalizamos una tarea, el desarrollador no realiza un merge directamente con la rama master. Sino que dicho de dicho desarrollador primeramente realiza un push de una rama feature al servidor central y envía un pull request para solicitar que se añadan sus cambios. Una vez aceptado el pull request, deberemos sincronizar nuestro master con la de GitHub. Seguidamente, realizaremos un merge de la rama feature sobre master, y subiremos la rama master a repositorio central.

3. Gitflow Workflow

GitflowWorkflow utiliza un repositorio central, sin embargo, los desarrolladores trabajan en sus ramas y las suben al repositorio de GitHub.

a. Historical branches
En este sistema existen dos ramas principales: master y develop. La primera se utiliza para registrar la historia, mientras que la segunda es para incorporar cambios del resto de ramas feature.

b. Feature branches
Cada cambio que realicemos en el proyecto debería tener su propia rama, sin embargo, estas ramas denominadas feature, usarán la rama develop en vez de la master. En otras palabras, features nunca deberán interactuar de manera directa con la rama master de GitHub.

c. Release branches
Una vez que la rama develop ha adquirido un número suficiente de cambios, bifurcamos dicha rama mediante un fork. En la rama nueva creada denominada release, no podremos crear más cambios, únicamente tareas para orientar. Una vez finalizado, si que podrá unirse a la rama master y además a la rama develop.

d. Maintenance o Hotfix Branches
Las ramas de mantenimiento o hotfix, como su propio nombre indica, son ramas que se bifurcan directamente de la rama master y que se usan para reparar problemas. Cuando el problema se haya solucionado, esta rama está obligada a hacer un merge hacia la rama master y la develop.

4. Forking Workflow

Forking Workflow es un sistema en el que no se una un único servidor, sino que existe un repositorio original y de ese, los que se hayan bifurcado. Básicamente hacer un fork es realizar una copia de un proyecto, pero con un link hacia el original.
En este sentido, cada desarrollador podrá tener su propio repositorio en GitHub. Sin embargo, únicamente el que manetiene el proyecto original podrá hacer commits en el repositorio oficial.

El primer paso, hay que hacer un fork del repositorio oficial para crear una copia en el servidor. Posteriormente, el desarrollador clonará el repositorio para obtener una copia en el repositorio local. Una vez que se hace un commit, este se realiza en el servidor del propio desarrollador y no en el oficial. Para que los cambios puedan mostrarse en el oficial, deberemos enviar un pull request.
